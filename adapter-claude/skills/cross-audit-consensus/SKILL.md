# /cross-audit-consensus — 三方对抗结果汇总

## 触发场景

- 用户说："汇总三方 / 比对三方 / 出共识表"
- `/cross-audit` 已产出三方独立报告，需要 Claude 充当 reviewer 折叠
- 研究归档、文档审计等三方完成后

## 输出格式（强制）

### 1. 三方完成度速览

```markdown
| 方 | 状态 | 字数 | 关键发现数 |
|---|---|---|---|
| Claude | 完整 | ~4000 | 8 |
| Codex | 完整 | ~4500 | 11 |
| Gemini | 失败/偏短 | 1.3KB | 0 实质 |
```

如有方失败，必须**显式标注**（Gemini 在某些场景偏短/幻觉，不得静默忽略）。

### 2. 共识表

```markdown
| 发现/方案 | Claude | Codex | Gemini | 共识 | 证据 |
|---|---|---|---|---|---|
| [简短描述] | Critical | Critical | — | 2/3 | 文件:行号 / URL |
```

**裁决原则**：

- 2/3 共识默认采纳
- 1/3 独有发现进消费项目的决策备忘（不采纳但不丢）
- 0 共识 / 严重冲突 → 用户裁决（不缝合方案）

### 2.5 量化评分（强制，适用 PR 审查场景）

每次 consensus 必须给出三方各自在以下 4 维的打分（0-10），最后汇总为综合分。**这是评估不是裁决** —— 共识 / 冲突的采纳仍按 §2 原则，但分数决定"是否该 retry 一轮"。

```markdown
| 维度 | 含义 | Claude | Codex | Gemini |
|---|---|---|---|---|
| Scope Fidelity | 与 Issue / Sprint Contract 的 Touches / No-Touch / Interface Delta 吻合度 | | | |
| Code Quality | 消费项目 lint 违规、圈复杂度、架构一致（独立于 CI 的人工审查） | | | |
| Test Coverage | 新增 / 修改逻辑是否有对应测试，异常路径是否覆盖 | | | |
| Architecture Fit | 是否遵循消费项目的工程哲学（奥卡姆剃刀 / YAGNI / SSOT / 可逆性）与现有模式 | | | |
| 综合 | 上 4 项平均 | | | |
```

**Retry 阈值**：

- 任一方综合分 < 7.0 → consensus 必须给出"建议按该方反馈 retry 一轮"的显式建议
- 任一方综合分 < 5.0 → **硬性要求 retry**（继续合并视为违反"质疑方批准"纪律）
- 三方综合分均 ≥ 8.5 → consensus 可直接给出"立即合并"建议
- 三方综合分方差 > 2.5 → 必须在冲突表列出原因（为什么一方严重低分）

**评分原则**：

- 每维度 10 分制，整数打分；不打半分
- 打分依据必须引用具体证据（与 §2 共识表共用证据列）
- Sprint Contract 未填写的 Issue → Scope Fidelity 默认扣 2 分

**评分方权重（SSOT）**：

| 评分方 | 角色 | 裁决权重 |
|---|---|---|
| Codex（gpt-5.4 --effort high，对照 plan 明文逐条打分） | 主锚点 | 始终 |
| Gemini（守评分纪律：8.0 起评 + plan 锚点 + 方差自检） | 次锚点 | 守纪律才计入；不守纪律降为参考 |
| Gemini（不守评分纪律） | 参考 | 与 Claude 自审同级 |
| Claude（主对话内审自己刚实施的 PR） | 参考 | 不作锚点（实施方自审 bias） |
| Claude（主对话内审他人或历史 PR） | 次锚点 | 独立视角但仍弱于 Codex |
| Claude（通过独立 Agent 委派审自己 PR） | 次锚点 | Agent fork context 减弱 bias，但仍弱于 Codex |

**关键区分**：

- **主锚点** = Codex 一家；共识 / 冲突裁决以 Codex 评分为基准
- **次锚点** = 进入裁决计算但权重低于主锚点；方差 > 2.5 时以主锚点为准并列出冲突原因
- **参考** = 分数列出供人类阅读，不进入裁决计算

**为什么这样分层**：实战多次暴露 Claude 自审分虚高 vs Codex 严格对照 plan；Gemini 偏重"方向正确"也绕过 plan 明文逐条。只有 Codex 的严格评分是裁决的可靠基准。

**Retry 阈值与锚点的关系**：

- Retry 判定（< 7.0 建议 / < 5.0 硬性 / 方差 > 2.5 冲突）按**主锚点 Codex 分数**触发
- 次锚点和参考分不单独触发 retry，但会计入冲突表协助诊断根因

### 3. 单家独有发现归档

| 来源 | 发现 | 处置 |
|---|---|---|
| Claude 独有 | ... | 进决策备忘 / 推迟 / 拒绝 |
| Codex 独有 | ... | 同上 |
| Gemini 独有 | ... | 同上 |

### 4. 冲突点（如有）

```markdown
| 维度 | Claude 主张 | Codex 主张 | Gemini 主张 | 我推荐 | 证据 |
|---|---|---|---|---|---|
```

冲突裁决证据等级：编译 / 测试 / 日志结果 > 官方文档引用 > 代码分析推断 > 经验性判断。

### 5. 给用户的行动清单

- [ ] 高 ROI 共识项（建议立即采纳）
- [ ] 中等价值（用户裁决）
- [ ] 低价值 / 不采纳（理由）
- [ ] 冲突需用户拍板的具体问题

## 必须遵守的纪律

1. **同源**：汇总前确认三方输入是同一任务，不是分工的子集
2. **Gemini 验证**：Gemini 涉及具体文件 diff 的输出必须代码验证
3. **不缝合方案**：冲突时选一方完整方案，不混合（避免逻辑断裂）
4. **保留原始**：原始三方输出不删（落到 `~/.claude/plans/` 或消费项目归档目录）

## 输出位置

- 临时汇总：贴回 chat
- 决策类汇总：`~/.claude/plans/<topic>-summary.md` 或消费项目的 `<archive-dir>/<topic>-<date>.md`
- 影响 ADR 的 → 同步到消费项目的 ADR 对应段落

## 反模式

- 错：三方都没看就汇总（必须 Read 所有三方原始输出）
- 错：Gemini 失败时假装它有效（要标失败）
- 错：「三方都同意」但没列证据（必须给文件路径 / URL）
- 错：缝合三方方案制造逻辑断裂的混合体（选一方完整方案）
